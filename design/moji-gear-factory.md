# もじギア・ファクトリー（Moji-Gear Factory）— 設計・仕様書

> 新カテゴリ「**ことば**」の中核となる、**5〜7歳児むけ 読み書きの素地（emergent literacy）×
> 実行機能（executive function）** ゲーム。
> 既存の「ふしぎな だいとしょかん」（`MathAdventureUnit`）と同じ **としょかん型**（本＝ページを
> 順に解放）で、まず **10ページ（＝10の工場ライン）** を作る。

このドキュメントは「世界観」「問題案」「仕様書」をまとめたもの。**実装前の合意用**であり、
着手時は `authoring-problems` スキルと `math-adventure-architecture.md` を必ず参照する。

---

## 0. このドキュメントの結論（先に要点）

- 対象は **5〜7歳（年長〜小2）**。既存アプリの最年少 `baby`（2〜3歳）と `sansu`（小1〜）に対し、
  **ひらがなを読み始め〜だいたい読める時期**の「ことば・音韻の操作力」にねらいを定める。
- 学びの中核は **エビデンスが最も強い2つ**＝① 音韻認識（とくに **モーラ意識**）と
  ② 文字知識（かな-音の対応）。ここに ③ 実行機能（抑制・ワーキングメモリ・切り替え）の
  ギミックを **のせる**。①②が土台、③はやりがい・転移の上乗せ。
- 元コンセプト（IF-THEN／NOT逆唱／リソース管理＋タイマー）は魅力的だが、5〜7歳の
  認知負荷に照らして **2点を見直す**（§5）：
  1. **タイマー（時間プレッシャー）は原則なくす／ごく弱くする**。この年齢では不安が学習を妨げ、
     内省・自己修正の時間を奪う（逆効果のエビデンスがある）。
  2. **「分類＋音韻抽出＋文字選択」の3重課題を同時に課さない**。1ページ＝1つの新しい負荷に
     絞り、足場（scaffold）で段階的に重ねる。
- 10ページは **音韻認識の発達順序**（語→モーラ分解→語頭→語尾→同定→削除→逆唱）に
  文字知識・特殊音節を織り込み、**やさしい→むずかしい**に一直線に並べる（§7）。

## 2. エビデンス基盤（なぜ効くのか）

> 「エビデンスに基づいた効果のある設計」にするため、各ページのねらいは下の3本柱の
> どれか（できれば複数）に必ず紐づける。紐づかない“なんとなく楽しいだけ”の課題は入れない。

### 2-1. 音韻認識（Phonological Awareness）— 最重要の土台
- 読み習得を **最も強く予測** する就学前スキル。日本語は **モーラ（拍）** が基本単位
  （英語の音素とは違う）。「すいか」=「す・い・か」の3拍を、数える・取り出す・入れ替える力。
- 日本語かな習得研究（天野清ほか）が示す要点：**語を音韻的に分解する操作（音韻分析）が
  できるようになると、かな文字の習得が一気に進む**。文字を教える前後で音韻課題が伸びる相互作用。
- 発達のおおまかな順序（やさしい→むずかしい）：
  1. 語の気づき（これは「ひとつのことば」）
  2. **モーラ分解・数える**（「ねこ」は 2つ／手をたたいて数える）
  3. **語頭音の抽出**（「りんご」のさいしょは「り」）
  4. **語尾音の抽出**（「りんご」のおしりは「ご」）＝ しりとりの基礎
  5. **音の同定・マッチング**（「さいしょが おなじ」もの探し）
  6. **削除**（「いちご」から「い」をとると「ちご」）
  7. **逆唱・並べ替え**（「ねこ」→「こね」）＝ ワーキングメモリ最大負荷
- → これがそのまま **ページの難易度順** になる（§7）。

### 2-2. 文字知識（Letter / Grapheme Knowledge）
- かな1文字＝1モーラ（透明性が高い表音文字）。**文字-音の対応** を知っていることが、
  音韻認識と並ぶ強い予測因子。両者は **掛け算**（音は分かるが文字が出てこない／文字は読めるが
  音と結びつかない、のどちらが欠けても読めない）。
- ねらい：見て選ぶ → 並べて作る、の順で「音から文字を引き出す」回路を育てる。
- **特殊音節（濁音・半濁音・拗音・長音・促音）** はかな習得の難所。最後のページで扱う。

### 2-3. 実行機能（Executive Function）
- 就学前の実行機能は **学力全体（読み・算数）の予測因子**。3要素：
  - **抑制制御**（やりたくなる反応を止める）：例 Day-Night課題、ハト/カラス課題。
  - **ワーキングメモリ**（覚えながら操作する）：逆唱はこれを強く使う。
  - **認知的柔軟性／ルール切り替え**（DCCS, Zelazo）：「いま は あかルール → あおルール」の切替。
- 本ゲームの「ギア」「ルールカード」「IF-くん」は、この **ルール保持＋切り替え＋抑制** を
  遊びに落とすための装置。**ただし音韻・文字の土台を崩してまで足さない**（§5）。

### 2-4. 「効く」ための学習設計のエビデンス（実装に効く一般原則）
- **適応的難易度**：常に **正答率 ~80%** 前後（やさしすぎ＝退屈、難しすぎ＝無力感）に保つ。
- **時間プレッシャーは幼児には逆効果**になりやすい：不安が増え、内省・自己修正が減る
  → **タイマーは原則使わない／使うなら“演出”で、失敗ペナルティにしない**。
- **即時・前向きフィードバック**：正解は短く強く祝い、つまずきは「次に考えること」を示す
  （既存 `buildHint` 思想）。
- **足場かけ（scaffolding）とフェード**：最初は選択肢を絞る・ヒントを濃くする → できたら外す。
- **検索練習（retrieval）・間隔をあけた反復**：同じ語を別の課題で再会させる（語彙の定着）。
- **マルチメディア原則**：音声＋絵を同時に。文字が読めない子に文字だけで指示しない。

## 3. 設計原則（このゲーム固有のルール）

1. **出題は必ず「音声＋絵」**。文字は“答えを作るための部品”として出す（読めなくても遊べる）。
2. **1ページ＝1つの新しい認知負荷**。新スキルは1つずつ。前ページの力を前提に積む。
3. **タイマーで失敗させない**。急かす演出は最小限、ペナルティにしない（§5）。
4. **三重課題を避ける**。分類・音韻操作・文字選択を同時に全部やらせない。足場で重ねる。
5. **適応難易度**：直近の反応（正誤・迷い時間）で、文字数・選択肢数・ルール数を上下させる。
6. **ぜんぶ ひらがな・前向き文言**（既存の鉄則を踏襲）。「まちがい」と言わない。
7. **動的生成**：語＝「絵文字／音声／カテゴリ／モーラ列／正解文字」を持つ1データから、
   画面（選択肢・盤面）を組み立てる。デザイナーが画面を量産しない。
8. **検証はテストで守る**（§11）。「一意解」「足りない部品が無い」「正解が選択肢にある」等を自動検査。

## 4. 語データの中核（コンテンツの単位）

すべてのページが共有する **語の辞書（WordItem）** を1つ持つ。これが動的生成の燃料。

```ts
interface WordItem {
  id: string;          // 'suika'
  display: string;     // 絵文字 or 画像 '🍉'
  reading: string;     // 'すいか'（TTS用）
  mora: string[];      // ['す','い','か']  ← モーラ列（音韻操作の単位）
  category: WordCat;   // 'food' | 'animal' | 'vehicle' | 'nature' | 'thing'
  difficulty: 1|2|3;   // モーラ数・特殊音節の有無でだいたい決まる
  special?: ('voiced'|'semivoiced'|'youon'|'choon'|'sokuon')[]; // 濁音…等
}
```

- まず **絵文字で表現できる2〜4モーラの基本語 40〜60語** をシードする
  （食べ物・動物・乗り物・自然・もの）。`tts.ts` の `EMOJI_READINGS` を拡張して読みを足す。
- モーラ列はデータに**手で持つ**（「きゃ」「しゃ」等の拗音は1モーラとして1要素にする）。
  プログラムでの自動分解はミスるので、辞書で確定させる。
- 特殊音節（が/ぱ/きゃ/けー/がっこう…）は `special` で印をつけ、難所ページでだけ出す。

## 5. 元コンセプトの評価と見直し（エビデンス観点）

ユーザー提案の3ステージは「論理的ギミック」が魅力。**世界観とギア／IF-くんは全面採用**。
ただし5〜7歳の認知特性に照らし、次の2点を**見直す**（これが「効く設計」の肝）。

### 見直し① タイマー（時間プレッシャー）は外す／演出に格下げ
- 提案の Stage3「お掃除ルンバが迫る制限時間」は、**幼児には逆効果**になりやすい。
  時間圧は不安を生み、**内省・自己修正・プランニングの時間を奪う**（=ねらいである
  プランニングそのものを壊す）。
- 代替：**ルンバは“急かす存在”ではなく“ごほうびを運ぶ相棒”**にする。時間で失敗させず、
  「落ち着いて2つ作れたらルンバがピカピカに掃除してくれる」演出に。**プランニングは
  “手数（ステップ）”で評価**（最少ステップ＝ぴったり賞）し、時間では測らない。

### 見直し② 三重課題（分類＋音韻抽出＋文字選択）を同時にやらせない
- 提案 Stage1 は「すいか→食べ物→🔵ルール→おしり音『か』→文字『か』を探す」と、
  **分類・ルール適用・音韻抽出・文字探索の4工程**を一度に要求する。5〜7歳のワーキングメモリ
  （だいたい同時に2〜4要素）を超えやすく、**できない＝この子の力不足ではなく設計過負荷**。
- 代替：**工程を別ページに分解**して段階導入し、最後のページで“合体”させる。
  - 音韻抽出だけのページ（ルール無し）
  - ルール切り替えだけのページ（音韻は語頭固定）
  - 文字を“作る”ページ（音韻は提示済み）
  - …を経て、**最終ページ（IF-くん工場）で IF-THEN＋音韻＋文字を合体**（足場つき）。
- 逆唱（Stage2）は WM 最大負荷なので **2モーラ語からスタート**し、NOTゲートの視覚トリガーは
  **逆唱に慣れてから**足す（同時に出さない）。

> まとめ：**世界観とギミックは活かす。負荷の“出す順番”だけ、発達に合わせて組み替える。**

## 6. 世界観（Worldview）

> **詳細な世界観・トーン・視覚言語・キャラ・効果音・10ラインの衣装は
> `design/moji-gear-philosophy.md` にまとめた**（算数版の `adventure-philosophy.md` の対）。
> ここでは要約だけ。

まんまるロボ **IF-くん**を ことば（ねんりょう）で動かして、**いろんな せかい**を めぐる旅。
スピーカーから きこえる **「コトバ」** を **ギア（もじの はぐるま）** にして **カチッ**とはめると、
IF-くんが動き、**その世界が いろづく**。子どもは IF-くんの あいぼう。

- **芯は全ライン共通**：① IF-くん ② コトバが光の粒にほどける ③ 文字の歯車を拾って**カチッ**
  ④ はまると連鎖して回りIF-くんがよろこぶ。**ここは10ライン不変**。
- **変えるのは“どの世界か”**（単調回避の核）：10ラインは **別テーマの世界**＝どうぶつえん🦁・
  おかしのくに🍰・うちゅう🚀・うみ🌊・おばけやしき👻・のりもの🚒・にわ🌸・きょうりゅう🦕・
  サーカス🎪・IF-くん だいこうじょう🤖。**世界の色・住人・お話・ごほうびを総替え**する
  （算数版の多ゾーンと同じ変化の付け方）。
- **だいとしょかん的なハブ**：工房（ホーム）が **本棚＝旅の出発点**。最初の世界だけ行ける →
  クリアで次の世界が地図に出る（**直線アンロック・末尾追加**＝算数版とおなじ）。
- **キャラ**：相棒 **IF-くん** ＋ **世界ごとの ともだち**（くまの えんちょう・うちゅうじん・たこ…）。
  お邪魔キャラ **NOT-くん**（緑、ライン8〜）。ごほうびを運ぶ **ルンバ**（急かさない）。
- **おとな向けの言葉（コンパイル・エンジニア）は こども画面に出さない**（設計の比喩のみ）。

> 詳しい世界観・住人・色・効果音・10世界の衣装は **`design/moji-gear-philosophy.md`**。

## 7. 10ページ（10の製造ライン）— 問題案

> 並び順＝難易度順＝アンロック順（前ラインのクリアで次が点灯）。
> 各ラインは「**1つの新しい認知負荷**」に絞り、後半で実行機能ギミックを重ねる。
> 「エビデンス」列は §2 の柱（音韻=Phon / 文字=Letter / 実行機能=EF）への紐づけ。
> **「せかい」列＝テーマの衣装**（住人・色・お話）。学びは固定、見た目は世界ごとに総替え（§6）。
> 下の詳細はメカニクス中心。世界の衣装の詳細は `design/moji-gear-philosophy.md`。

| # | せかい（テーマ） | ねらい（1行） | エビデンス |
|---|---|---|---|
| 1 | 🦁 どうぶつえん | モーラを数える（語の分解） | Phon②（分解・計数） |
| 2 | 🍰 おかしのくに | 語頭音の抽出 | Phon③ / Letter |
| 3 | 🚀 うちゅう（しりとり） | 語尾音の抽出 | Phon④ / Letter |
| 4 | 🌊 うみ | 音の同定・マッチング | Phon⑤ |
| 5 | 👻 おばけやしき | 文字を順に並べて語を作る | Letter / Phon |
| 6 | 🚒 のりもの基地 | カテゴリで処理を切替（IF-THEN・1ルール） | EF切替 / Phon③④ |
| 7 | 🌸 ひみつの にわ | 音の削除 | Phon⑥（最難の音韻操作） |
| 8 | 🦕 きょうりゅうの しま（NOTくん） | 逆唱（2モーラ→3モーラ） | EF-WM / Phon⑦ |
| 9 | 🎪 サーカス | 特殊音節（濁音・半濁音・拗音・長音・促音） | Letter（難所） |

以下、各ラインの詳細。**問題例の「→」は子の頭の中の処理**。

### ライン1 🔢 いくつの ギア？（モーラ計数）
- **画面**：絵＋音声「すいか！」。下に空のギア枠が並ぶ。子は **手をたたく感覚**で
  「いくつの音？」をタップ（2/3/4 の数ボタン、または枠の数を選ぶ）。
- **ループ**：音声「すいか」→ 「す・い・か」を区切って再生（1拍ずつ光る）→ 「3つ！」を選ぶ。
- **足場**：最初は1拍ずつ TTS が区切って読む → 慣れたら続けて1回だけ読む。
- **適応**：2モーラ→3→4 と伸ばす。選択肢は近い数（2/3/4）。
- **エビデンス**：分解・計数は音韻認識の入り口。すべての後続ラインの土台。

### ライン2 🔤 さいしょの ギア（語頭音）
- **画面**：絵＋音声「りんご！」。文字ギアが4つ流れてくる（り/ご/す/ね 等）。
  「**さいしょの おと**」のギアをロボの胸にハメる。
- **ループ**：「りんご」→ 語頭「り」→ 文字「り」をタップ。
- **適応**：ダミーは「語中・語尾の文字」を混ぜる（語頭以外を抑制＝EF抑制も少し働く）。
  難化で「さいしょが似てる」語（り/ら）を混ぜる。
- **エビデンス**：語頭音抽出＋文字対応。読みの中核回路。

### ライン3 🔚 おしりの ギア（語尾音・しりとり）
- **画面**：しりとり風。IF-くんが「**…ご**！」と最後の音を欲しがる。「りんご」のおしり「ご」を探す。
- **ループ**：「りんご」→ 語尾「ご」→ 文字「ご」。発展：次の語の語頭が前の語尾と繋がる演出。
- **適応**：2→3→4モーラ。難化で「しりとりチェーン」（前の語尾＝次の語頭）に。
- **エビデンス**：語尾抽出は語頭より難しい。しりとり＝日本語音韻遊びの王道。

### ライン4 🔎 おなじ おとの なかま（音の同定）
- **画面**：見本「さかな」。下に3つの絵。「**さいしょが おなじ**」もの（さくらんぼ）を選ぶ。
- **ループ**：見本の語頭「さ」を保持 → 候補の語頭と照合 → 一致を選ぶ（WM＋照合）。
- **適応**：語頭→語尾の同定へ。候補数 3→4。
- **エビデンス**：音の同定・分類。保持しながら比べる＝WMも働く。

### ライン5 🧩 ギアを ならべて コトバ（文字合成）
- **画面**：音声「ねこ！」。バラの文字ギア（ね/こ/た/い）。**順番に**ハメて「ねこ」を作る。
- **ループ**：「ねこ」→「ね」→「こ」の順でドラッグ/タップ。@dnd-kit が使える。
- **足場**：最初は枠に薄く正解文字が下書き → フェードで消す。
- **適応**：2→3→4モーラ。余分なダミー文字を増やす。
- **エビデンス**：音から文字列を“産出”する（再認より難しい）。読み書き両方の素地。

### ライン6 🔴🔵 ルールカード工場（IF-THEN・1ルール切替）
- **画面**：上部にルールカード。
  - 🔴 **もし【たべもの】なら → さいしょの ギア**
  - 🔵 **もし【いきもの】なら → おしりの ギア**
- **ループ**：「すいか」→ たべもの → 🔴 → 語頭「す」。「ねこ」→ いきもの → 🔵 → 語尾「こ」。
- **負荷管理**：**音韻操作は ライン2/3 で習得済みのものだけ**を使う（新規負荷はルール切替のみ）。
  カテゴリは絵で明確なものに限定（食べ物 vs 動物）。最初は1ルール、慣れたら2ルール。
- **適応**：ルール1個→2個。カテゴリ2種→3種。
- **エビデンス**：DCCS 的なルール切替（認知的柔軟性）＋抑制。**§5見直し②**に従い三重課題化を回避。

### ライン7 ✂️ ギアを ひとつ ぬく（音の削除）
- **画面**：「いちご」のギア列から **さいしょのギア「い」をぬく** と何になる？→「ちご」。
- **ループ**：「いちご」→「い」を取る →「ちご」を音で確認 → 残りの文字列/絵を選ぶ。
- **足場**：抜いた瞬間に TTS で「ちご」と読み上げ（自己確認）。
- **適応**：語頭削除→語尾削除。3→4モーラ。
- **エビデンス**：削除は音韻操作の中でも高難度（保持しながら操作）。

### ライン8 🔁 さかさま ギア（逆唱・NOTくん）
- **画面**：緑の **NOT-くん** が出たら「**さかさま**にコンパイル」。「ねこ」→「こね」。
- **ループ**：NOT-くん登場 → さかさまルール → 「ねこ」を反転「こね」→ こ・ね を順にハメる。
- **負荷管理**：**2モーラ語から**。逆唱に慣れるまで NOT-くんは“常時ON”（条件分岐なし）。
  慣れたら「NOT-くんが居る時だけ反転」の視覚トリガー（抑制＋切替）を足す（§5見直し②）。
- **適応**：2→3モーラ。条件トリガーの有無。
- **エビデンス**：逆唱＝WM最大負荷の音韻操作。幼児の WM 容量を広げる強い課題。

### ライン9 ✨ とくべつ ギア（特殊音節）
- **画面**：濁音・半濁音・拗音・長音・促音。「ぱんだ」の「ぱ」、「きゃべつ」の「きゃ」、
  「けーき」の長音、「がっこう」の促音。`special` 印の語だけ出す。
- **ループ**：語頭/語尾の特殊音を選ぶ。拗音は「きゃ」を1ギアとして扱う。
- **足場**：清音と並べて対比（か↔が、つ↔っ）。読み上げでゆっくり差を聞かせる。
- **適応**：濁音→半濁音→長音→拗音→促音 の順（習得難度順）。
- **エビデンス**：特殊音節はかな習得の最大の難所。最後に集中的に扱う。

### ライン10 🤖 IF-くん だいこうじょう（合体・ボス）
- **画面**：これまでの工程を**合体**。ルールカード（IF-THEN）＋音韻操作＋文字合成を1問で。
  例：🔴たべもの→語頭ギアを作る／🔵いきもの→語尾ギアを作る、を**2語連続**で。
- **ループ（プランニング）**：手持ちギア（た/こ/い/ぬ）で「いぬ」「たこ」を**最少ステップ**で
  作る順を考える（提案 Stage3 のプランニング。**ただしタイマー無し**＝§5見直し①）。
- **足場**：最初はルール1個＋1語 → 2ルール＋2語へ。足場プリフィル（一部ギアを置いておく）も可。
- **適応**：語数 1→2、ルール 1→2、足場あり→なし。
- **エビデンス**：実行機能の総合（保持・切替・抑制・プランニング）＋音韻＋文字の統合。
  「大人が見ても“論理を使っている”」手応えのボス。**最少ステップ＝ぴったり賞**で評価。

## 8. 適応的難易度とログ（“効く”を支える裏側）

### 調整するパラメータ（ページ共通の難易度ノブ）
- **モーラ数**（2→3→4）
- **選択肢数／ダミー数**（少→多）
- **ルール数**（1→2）・**カテゴリ種**（2→3）
- **足場の強さ**（下書き表示・1拍区切り読み → フェード）
- **特殊音節の有無**

### 適応ロジック（簡易・実装しやすい形から）
- まず **連続正答／連続つまずきの単純ルール**で1段ずつ上下（例：3連続正解で+1、2連続つまずきで−1）。
  これだけでも ~80% 正答帯に寄せられる。IRT 等の本格モデルは将来拡張（過剰実装しない）。
- **迷い時間（latency）** と **特殊音節別のつまずき** をログに残し、苦手な音（濁音/拗音等）を
  少し多めに再出題（検索練習）。**latency で減点はしない**（§5：時間で罰しない）。

### ログに残すもの（パーソナライズと検証の燃料）
- ライン別／モーラ数別の正答率・反応時間・つまずいた `special` の種類・使ったヒント段階。
- `storage.ts`（localStorage）に保存。既存の冒険進捗（`math-adventure:` 名前空間）にならい
  **`kotoba-adventure:` 名前空間**で別管理する。

### 問題の型（提案）

```ts
  hintData?: unknown;        // buildHint 用（前向き文言を段階生成）
}
```

## 10. 検証（テストで守る）

## 13. 10より さきの 拡張設計（世界を ずっと足せるように）

> 「10 でおわり」ではなく、**11・12・13… と あとから 世界を 足しつづけられる** 構造にする。
> 算数版が としょかんに ゾーンを 何十も足せているのと同じ仕組みを 最初から 用意しておく。

### 拡張の核：**世界＝「テーマ」＋「既存メカニクスの組み合わせ」**

新しい世界を足すのに **新しいゲーム（メカニクス）を作る必要はない**。
世界1〜10 で 作った **10 個のメカニクス（`lineId`）を 部品として 組み合わせる**だけ。

```ts
interface WorldDef {
  id: string;            // 'world-castle'
  name: string;          // 'なぞなぞ アイランド'
  emoji: string;         // '🏝️'
  friend: string;        // 'なぞなぞマスター'（住人）
  story: string;         // 入口の2〜3行（ひらがな）
  tint: string;          // 世界の色（背景・地図のにじみ）
  lineIds: LineId[];     // ★ ここに 既存メカニクスを 1つ以上 入れる（プール）
  difficulty?: {         // 任意：この世界の難しさノブ（§8のノブを上書き）
    minMora?: number; maxMora?: number;
    ruleCount?: number; special?: boolean;
  };
}
```

- **世界1〜10**：`lineIds` が **1つ**（＝1メカニクスを 順に 教える「教材」）。
- **世界11以降**：`lineIds` が **複数**（＝覚えた技を 混ぜて出す「腕試し・応用」）。
  これは算数版 `zones.ts` の `unitIds: string[]`（プール）と **まったく同じ思想**。
  マップの各ノードで `lineIds` から ランダムに 1つ選んで出題する。

### 世界を足す手順（チェックリスト）
1. `kotoba/zones.ts` の **配列末尾**に `WorldDef` を1件追加（**途中挿入しない**＝直線アンロックが
   巻き戻る。算数版の既知の罠）。
2. `lineIds` は **既存のものを使う**＝アダプター追加 **不要**（`adapter not found` も起きない）。
3. テーマ用の `tint`／住人の絵文字だけ足す（UIの新規実装は基本ゼロ）。
4. テスト（`zone adapter coverage`・マップ到達可能性）を流す＝自動で守られる。
5. **本当に新しい遊びを足したいときだけ** 新 `lineId`＋アダプター＋専用テストを追加（まれ）。

### 解放のしかた
- 世界10（IF-くん だいこうじょう＝ボス）**クリア後に 11以降が 地図に出る**。
  位置づけは「**ボスのあとに ひらく 上級の せかい**」（算数版の“理数センス本”と同じ設計）。
- やはり **直線アンロック・末尾追加**。既存セーブを壊さない。

### まず作る続き3世界（11〜13）— 既存メカニクスの組み合わせ

| # | せかい | 住人 | 組み合わせ（`lineIds`）| ねらい・難度 |
|---|---|---|---|---|
| 11 | 🏝️ なぞなぞ アイランド | なぞなぞマスター | `count-mora`+`first-mora`+`last-mora`+`match-sound` | **基礎の腕試し**（1〜4の復習ミックス）。3〜4モーラ中心 |
| 12 | 🎡 ことばの ゆうえんち | あんない係ロボ | `match-sound`+`build-word`+`delete-mora` | **中級ミックス**（同定・合成・削除）。アトラクションごとに技が変わる |
| 13 | 🌌 さかさま タワー | 改心した NOT-くん | `reverse-word`+`special-mora`+`rule-card` | **上級ミックス**（逆唱・特殊音節・IF-THEN）。最難。長い逆唱＋濁音/拗音 |

- **11 なぞなぞ アイランド**：宝の地図のなぞなぞを解く体。例「これ なーんだ？🐘」→
  音声「ぞうさん」→（その問いが count なら）「いくつの音？」、first なら「さいしょは？」…と
  `lineIds` から 出る技が 毎回 変わる＝**飽きない復習**。やさしめ（ボス直後の自信回復）。
- **12 ことばの ゆうえんち**：観覧車＝同定、コースター＝文字合成、おばけ屋敷＝削除…と
  **アトラクション＝メカニクス**に見立てる。中級の技を 混ぜて 維持・定着（検索練習）。
- **13 さかさま タワー**：NOT-くんの本拠地（改心して 案内役に）。逆唱を 3モーラに伸ばし、
  特殊音節（濁音・拗音）を 含む語、IF-THEN の 2ルールを 混ぜる＝**5〜7歳の上限の腕試し**。
  クリアで NOT-くんが 完全に 仲間になる演出。

> この3つは いずれも **新メカニクス ゼロ**＝§9の `MojiQuestion`／既存アダプターで そのまま出せる。
> 「世界を足す」が データ追加だけで済むことの 実例＆ひな型にする。世界14以降も同じ作法で増やす。

### 拡張その2（2026-06-16）：語ふやし＋難易度すこし上げ＋世界14〜18

ユーザー要望「ゾーンを増やす／語を多めに／難易度を少し上げる」をエビデンスに沿って実装。

- **語を約28語追加**（計約78語）。3〜4（一部5）モーラ・特殊音を厚めに。
  → **狙い**：音韻スキルは特定語の暗記でなく **般化（transfer）** が本体。語彙が増えるほど
  「初見の語でも音を操作できる」力になる（語が少ないと丸暗記で学びが痩せる）。
- **適応レベル4を追加**（`adaptive.ts`：4〜5モーラ・5択）。**床（レベル1）は据え置き、天井だけ伸ばす**。
  → **狙い**：学習は ~80% 正答帯（最近接発達領域）が最も伸びる。入口を難しくすると脱落するので、
  **適応の上限だけ拡張**して「できる子はもっと上へ、苦手な子はやさしいまま」を両立。
- **新メカ2つ**：`middle-mora`（まんなかの音＝**語中音**）と `rhyme-match`（おしりが同じ＝**押韻**）。
  → **狙い**：語中音の抽出は語頭・語尾より発達的に難しく、**自然な難易度の一段**。押韻はしりとりと
  別系統の音韻感覚。どちらも「1ゾーン＝1つの新しい負荷」を守れる。
- **新ゾーン5つ**（計18世界）：こだまのたに🏔️(語中音)・おとあわせホール🎵(押韻)・
  やたいのよこちょう🍜(語頭/語尾/語中/押韻ミックス)・ドラゴンのすみか🐉(合成/削除/逆唱ミックス)・
  ことばのしろ👑(全部入りボス)。`WorldDef.startLevel` で **上級ゾーンは適応開始レベルを2〜3に**。
  → **狙い**：技を別文脈で混ぜて再会させる **インターリービング／間隔反復** は、まとめ練習より
  記憶定着が良い（頑健なエビデンス）。混合ゾーンは水増しでなく学習的に正当。
- **守ったこと**：タイマー（時間圧）は入れない／新負荷を一度に複数足さない（足場で1つずつ）。

### 拡張その3（2026-06-17）：ながい ことば＋「○ばんめ」＋世界19〜24

ユーザー要望「お題に文字数の多い語を／さいしょ・おしりだけでなく『何文字め』も／おすすめで6ゾーン」を実装。

- **5〜6モーラの語を約21語追加**（きょうりゅう・はりねずみ・ぱいなっぷる・はんばーがー・きゅうきゅうしゃ等）。
  → **狙い**：従来は2モーラ語（くり・とら）に偏っていた。長語は count/build/位置抽出の難度を自然に上げ、
  特殊音（促音・長音・拗音）も厚く乗る。音韻スキルの本体は**般化**なので語彙を増やすほど効く。
- **新メカ `nth-mora`（○ばんめの おと）**：first/last/middle の一般化。位置を `MojiQuestion.highlightIndex` で
  光の粒に番号＋リングで示す（「答え＝どの音か」は伏せ、「問い＝どの位置か」だけ見せる＝答えバレしない）。
  → **狙い**：位置を数えて抽出する力（音韻分析＋ワーキングメモリ）。middle より柔軟な難易度の一段。
- **新ゾーン6つ**（計24世界）：ナンバーこうじょう🔢(nth教材)・ながーいれっしゃ🚃(長語count)・
  おおきなくじら🐋(長語first/last)・メリーゴーランド🎠(長語build)・ことばたんてい🕵️(位置ミックス)・
  ことばのだいまおう🐲(長語の総合)。`WorldDef.difficulty.minMora/maxMora` で長語を強制。
- **罠（修正した）**：`MojiGearUnit.nextQuestion` のマージが `{...w.difficulty, ...a.opts()}` で、
  **adaptive が world.difficulty の minMora/maxMora を上書きしていた**＝design §13 が謳う「difficulty は
  §8ノブを上書き」が逆向きで、長語ゾーンを足しても短語しか出なかった。`{...a.opts(), ...w.difficulty}` に
  直して difficulty を優先（既存ゾーンは difficulty に mora 範囲を持たず special のみ＝無影響）。
  テスト `minMora/maxMora を 5〜6 にすると ながい語だけ 出る` で プール枯渇フォールバックも 検知する。
