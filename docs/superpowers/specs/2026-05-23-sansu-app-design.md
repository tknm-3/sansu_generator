# 子ども向け算数アプリ 設計仕様書

作成日: 2026-05-23

## 1. 概要

幼稚園年長〜小学校3年（5〜9歳）を対象にした、楽しく視覚的に算数を学べる無料Webアプリ。
算数の「仕組み」を絵とアニメで直感的に理解し、問題を自分で作って親に出すことで理解力を伸ばす。
相棒キャラと一緒に、スタンプ・成長・コレクションで「またやりたい」を生む。

### 対象ユーザー
- 主：幼稚園年長〜小学校3年の子ども（ひらがなは読める前提）
- 副：一緒に遊ぶ保護者（同じ端末・同じ画面で参加）

### 設計の前提・制約
- **完全無料**で公開・運用できること（サーバー維持費ゼロ）
- **アカウント不要**。親子は同じ端末・同じ画面で使う
- **タブレット横向き**を主対象（大きなタップ・ドラッグUI）。他端末でも動作する
- データはその端末内（localStorage）に保存。サーバー送信なし（プライバシー安全）
- 叱らない設計：減点・×・煽りはなし。ポジティブ報酬のみ

## 2. 技術スタック・アーキテクチャ

**構成：フロントエンドのみの静的Webアプリ（サーバー・DBなし）**

- フレームワーク：React + Vite + TypeScript
- スタイル：Tailwind CSS
- アニメーション：Framer Motion
- ドラッグ操作：dnd-kit
- 効果音・BGM：Howler.js
- 紙吹雪などの演出：canvas-confetti
- 読み上げ：Web Speech API（ブラウザ標準・無料TTS）
- データ保存：localStorage
- ホスティング：Cloudflare Pages（または GitHub Pages）— 完全無料の静的ホスティング

### 特性
- サーバーなしで全機能がブラウザ内で完結
- 初回読み込み後はオフラインでも動作可能（将来PWA化も容易）
- どの端末でも実行できる（モダンブラウザがあればよい）
- 進捗データは端末・ブラウザごと（端末間同期はしない。必要になれば将来エクスポート/インポートやバックエンド同期を検討＝現時点ではYAGNI）

### ゲームライクな表現について
本格的なゲームエンジンは不要。上記の軽量ライブラリ（Framer Motion / dnd-kit / Howler.js / canvas-confetti / Web Speech API）でアニメ・ドラッグ・効果音・ごほうび演出を実現する。将来より強い演出が必要な画面のみ PixiJS 等を部分的に追加可能。

## 3. モード構成

アプリは3つのモードと、共通のごほうび/コレクション要素で構成される。

```
ホーム（モード選択）
 ├─ ① がくしゅうモード（概念理解）
 ├─ ② もんだいチャレンジモード（文章題を解く）
 ├─ ③ もんだいづくりモード（問題を作って親に出す／保存）
 └─ なかま図鑑・スタンプ帳・きせかえ（ごほうび/コレクション）
```

相棒キャラは全モードを横断して登場する（後述）。

### 3.1 ① がくしゅうモード（概念理解）

**目的**：算数の仕組みを、相棒キャラとストーリー形式で視覚的に理解する。

**画面の流れ**
```
トピック選択 → ストーリー導入 → インタラクティブ問題 → 正解演出 → 次へ / スタンプ獲得
```

**インタラクションスタイル**：キャラクター＆ストーリー型。動物キャラが吹き出しで問題を出し、タップで読み上げ（TTS）。果物・ブロック・動物などの絵をタップ/ドラッグで操作して概念を発見する。

**単元リスト（難易度レベルで整理）**
- 年長〜小1：かずをかぞえる ／ 10をつくる（2つ・3つの数）／ たしざん ／ ひきざん
- 小2〜小3：二桁のたしざん（くりあがり）／ 二桁のひきざん（くりさがり）／ かけ算のしくみ ／ わり算のしくみ

**各単元の視覚的な仕組み**
- **10をつくる**：10マスフレームに果物を入れる。出題例「🍎が3こ。あと なんこで10こ？」→ 空きマスが点滅、答えボタン選択 or ドラッグで埋める。3つの数（例 3+3+4=10）にも対応
- **たしざん/ひきざん**：左右の動物が実際に動いて合流／去る
- **くりあがり（さくらんぼ計算）**：数の分解を1ステップずつ表示
  - 例 8+5：STEP1「8はあと2で10」→ STEP2「5を2と3に分ける（さくらんぼの枝で視覚化）」→ STEP3「8+2=10（10マス満タン）」→ STEP4「10+3=13」
  - 各STEPはタップで順番に出る
- **二桁のたしざん**：位ごとに分け、一桁で覚えた「さくらんぼで10をつくる」を一の位に応用
  - 例 28+15：STEP1「位に分ける（20と8／10と5）」→ STEP2「一の位 8+5 をさくらんぼで10をつくる→1束くりあがり、残り3」→ STEP3「十の位 20+10+くりあがり10＝40」→ STEP4「40+3＝43」
  - **くりあがりの「つながり」を強調**：一の位でできた10が1束に合体し、その同じ束が一の位から十の位へスーッと飛んでいくアニメで「10がひとつ上の位になる」を目で追える。流れを「①10こそろう→②1束に合体→③十の位へひっこし→④十の位が1束ふえる」のステップで明示
- **くりさがり**：十の位の1束をバラ10個に崩して一の位へ移す動き

### 3.2 ② もんだいチャレンジモード（文章題を解く）

**目的**：いろんな文章題が出題され、子どもが絵を操作して解く。スタンプ獲得。

**特徴**
- 相棒キャラが出題＋読み上げ
- 絵を操作して解く（例：タイヤを4こずつ車にドラッグ→車が完成→台数を答える）
- あまりは配り終えた残りとして絵で見える

②と③は**同じ「問題タイプ定義」を共有**する（作れる問題は答えるモードにも出せる）。

### 3.3 ③ もんだいづくりモード（問題作成）

**目的**：子どもがテンプレート穴埋めで文章題を作り、同じ画面で親が解く。作ることで理解力を上げる。

**画面の流れ**
```
テンプレート選択 → 穴埋め（数・絵・名前を選ぶ） → プレビュー（読み上げ確認）
 → 「ママ・パパにちょうせん！」 → 親が解答 → 子どもが◯つけ → スタンプ
```

**特徴**
- 数字はルーレット/ボタンで選択、絵柄はアイコン一覧から選択
- **答えは自動計算**（子どもは式を作るだけ）
- できた問題はキャラが読み上げ、子どもが確定
- 作った問題は「マイもんだいしゅう」に保存（localStorage）

### 3.4 学習設計：抽象（式）への橋渡しと復習

教育研究（CRA：Concrete→Representational→Abstract、および間隔反復による定着）にもとづく、学習効果を高めるための横断的な設計。

**抽象（式・数字）への橋渡し（CRAのフェードアウト）**
- 具体物（果物・タイヤ）と表象（10マス・束ブロック・さくらんぼ）の操作の**横に、常に対応する式を併記**する（例：操作と同時に「8 + 5 = 13」が育っていく）
- 習熟が進むと**具体物の支援を段階的に減らし**、最終的に式・数字だけで解く段階へ移行する（concrete → representational → abstract を意図的につなぐ）
- 各単元に「具体で理解 → 表象で確認 → 式だけで解く」の難易度ステップを持たせる

**復習・定着（間隔反復）**
- **スキル別の習熟度を記録**（後述データ設計の `mastery`）
- もんだいチャレンジモードに**過去に学んだ単元の復習を一定割合ミックス**して出題する
- 正答率が低い・しばらく出ていないスキルを**優先的に再出題**する（軽量な間隔反復）。間違えた問題は時間をおいて再登場させる
- 習熟度はマイルストーン報酬の根拠にもなり（5.0の「有能感のフィードバック」と接続）、孤立した外的報酬を避ける

## 4. 問題タイプ・コンテンツ

②③で共有する問題タイプのカタログ。すべて絵で操作して答えられるようにする。

### 問題タイプ
- **たしざん・ひきざん**：「🍎が3こ、2こふえると？」「🍪10こ、4こたべたら？」
- **かけ算（まとまり）**：「車3台、1台にタイヤ4こ。ぜんぶで何こ？」／「クモの足は8本。3匹で何本？」／「たまご1パック6こ、3パックで？」
- **わり算・包含除（◯個で何個つくれる？）**：「タイヤ20こで車何台？（20÷4＝5）」／「いちご3こでケーキ1こ。12こでケーキ何こ？」／「花を5本ずつの花束に。30本で何束？」
- **わり算・等分除**：「クッキー12こを3人で分けると1人何こ？」
- **あまりのあるわり算**：「ばんそうこう10こを4人に1こずつ。1人何こ？何こあまる？（10÷4＝2あまり2）」
- **切り上げて考える**：「20人がバスに乗る。1台6人。バス何台いる？（20÷6＝3あまり2→答え4台）」／「クッキー13まいを1皿4まいずつ。皿何枚いる？」
- **はんぶんこ・等分の入口**：「あめ8こを2人ではんぶんこ。1人何こ？」／「あめ7こをはんぶんこ…1こあまる、どうする？」
- **おかね・おつり**：「100円で30円のあめは何こ買える？おつりは？（100÷30＝3こ、おつり10円）」
- **くらべる（差）**：「あかいリボン12cm、あおいリボン8cm。どっちが何cm長い？」

### 重点とねらい
**はんぶんこ・あまり・切り上げ**を重点に置く。これらは「等しく分ける」「割り切れない時どうするか」という、分数・割合・概数につながる**基礎の考え方**を育てる。特に「切り上げて考える」問題（バス何台いる？等）は機械的な計算を超えた思考力を育てる。

### フレーバー文（飽きさせない味付け）
問題タイプのデータに**「フレーバー文」フィールド**を持たせ、ちょっと面白い言い回しでバリエーションを出す。
- 「きょうりゅうがクッキーを13こもってきた！4人でわけると1人何こ？何こあまる？（あまりはきょうりゅうが食べちゃう🦖）」
- 「おばけが20人でてきた！1台のバスに6人。バス何台よべばいい？」
- 「ねこがくつしたをかくした😼 8そくのうち2そくきえた。のこりは何そく？」
- 「あめが7こ。きみとあいぼうではんぶんこ…あれ？1こあまっちゃう。どうする？」

## 5. ごほうび・キャラ成長・コレクション

スタンプ・成長・コレクションの3層でリピートを生む。減点や煽りはなし、ポジティブ報酬のみ。

### 5.0 報酬設計の原則（アンダーマイニング効果の回避）
教育研究では、期待される外的報酬（シール・解放など）が子どもの内発的動機をかえって下げる「アンダーマイニング（過正当化）効果」が知られ、影響は大人より子どもで深刻。新奇性が切れると効果も落ちる。本アプリは外的報酬に依存しすぎないよう、次の原則を守る。
- **報酬は「行動の制御」ではなく「有能感のフィードバック」に寄せる**：「やったらシール」ではなく「できるようになったね」を可視化する位置づけにする
- **全問正解ごとに報酬を出さない**：報酬は意味あるマイルストーン（単元修了・習熟到達など）に絞る
- **ごほうび演出は任意（オフ可）**：演出を切っても遊べる。すでに意欲のある子の体験を壊さない
- **内発的フックを主軸に**：面白いフレーバー文・好奇心・上達の実感を中心の動機にし、外的報酬は補助とする
- **報酬演出は解答の最中ではなく完了後**に出し、学習中の集中を妨げない

### 5.1 スタンプ（達成の記録）
- 単元クリア・問題正解・問題作成でスタンプ獲得 → スタンプ帳に貯まる
- 「今日のスタンプ」「ぜんぶで何個」を見える化

### 5.2 相棒キャラ：成長＆交代
- **複数集められる**：マイルストーン達成・スタンプ数・各モードクリアで新しいなかまが解放（解放前は図鑑にシルエット表示）
- **いつでも交代できる**：「なかまをえらぶ」画面でアクティブな相棒をワンタップ切替（全モードに反映）
- **キャラごとに名前をつけられる**（改名も可）
- **成長は2層**
  - ぜんたいレベル：スタンプ＝経験値でアカウント全体のレベルが上がる（解放のカギ）
  - なかよし度（キャラごと）：アクティブにして一緒に遊ぶとそのキャラのなかよし度が上がり、見た目（例 たまご→ひよこ→成長）やセリフが増える → いろんな子を育てたくなる＝交代の動機

### 5.3 着せ替えコレクション
- ぼうし・リボン・メダルなどのアイテムを集め、どのキャラにも着せられる（自分だけの相棒に）

## 6. データ設計（localStorage）

サーバーなし。すべて端末内に保存。

```
math-app:profile     → { activeCharId, totalLevel, totalExp }
math-app:characters  → [ { id, name, affection, stage, equippedItems[] } ]  // 集めた相棒
math-app:collection  → { unlockedCharacterIds[], unlockedItemIds[] }
math-app:progress    → { 単元クリア状況, 各モード達成数 }
math-app:mastery     → { skillId: { 正答数, 誤答数, 最終出題日時, 習熟レベル } }  // スキル別習熟度（復習・間隔反復・有能感フィードバックの根拠）
math-app:stamps      → { 総数, 履歴[] }
math-app:myProblems  → [ 子どもが作った問題のリスト ]
```

将来：エクスポート/インポートでデータ移行できる余地を残す。

## 7. エラー処理・配慮

### 子ども向けの配慮（叱らない）
- **不正解**：×・減点なし。「おしい！もういちど」とキャラが優しく促す。続けて間違えたらヒント（対象がふわっと光る／STEP分解を見せる）
- **誤操作・連打**：大きなボタン・誤タップに強いUI。重要操作（出題・リセット）は確認を挟む
- **入力**：数字は選択式中心で入力ミスが起きにくい

### 技術的なエラー処理（優雅な劣化）
- **localStorage が使えない/壊れている**：初期状態で起動し復旧を試みる。保存失敗してもアプリは落とさない（プレイ継続優先）
- **TTS非対応ブラウザ**：読み上げボタンは出すが、無くても文字で遊べる
- **画像/音声の読み込み失敗**：代替表示・無音でも進行可能

## 8. テスト方針

- **算数ロジックは純粋関数＋ユニットテスト**：答え計算・あまり・切り上げ・解放判定を網羅（境界値：0、割り切れる、あまりあり等）。TDD（テスト先行）でロジックを固めてから演出を載せる
- **コンポーネントテスト**：各モードの主要操作（ドラッグで10をつくる、配ってあまりを出す等）
- **手動確認**：実機タブレットでタッチ操作・音・アニメを確認

## 9. 設計上の分離方針（保守性）

- **単元データ・問題タイプ・テンプレート**はコードと分離したデータ定義（穴の種類・式・絵柄候補・フレーバー文）として持つ → 追加が容易
- **算数の答え計算**は問題タイプ別の純粋関数に分離 → テストしやすい
- **報酬付与・解放判定**（スタンプ→経験値→レベル→解放）は1か所の純粋関数＋データ定義にまとめる
- **習熟度更新・復習出題の選定**（間隔反復：どのスキルを次に出すか）は純粋関数に分離し、境界値も含めてテストする
- **localStorageアクセス**はストレージ層を1モジュールに集約（読み書き一元管理）
- **演出（アニメ・音）**は共通コンポーネント化し全モードで再利用
