# プルリクを出すときの手順（pull-request-flow）

> 目的: **コンフリクトを残したまま PR を出さない**。レビュー前に必ずベース（`origin/main`）と
> 同期し、テスト・ビルドを緑にしてから出す。CLAUDE.md「プルリク（PR）を出すときの方針」の実務版。

## 鉄則
1. **PR を出す/更新する前に、ベースブランチとのコンフリクトを必ず解消する**（下の手順）。
2. コンフリクト解消後は **`npm run test` と `npm run build` を両方とおす**（build は `tsc` が走り、
   未使用定数 TS6133 などテストでは出ないエラーを拾う）。
3. **PR はユーザーが明示的に頼んだときだけ作る**。それ以外は作らない。
4. ベースは基本 `origin/main`（このリポジトリの既定）。PR の base もそこに合わせる。

## コンフリクト解消の手順（PR を出す前に毎回）
```bash
git fetch origin                               # 最新の origin/main を取る
git merge --no-commit --no-ff origin/main      # ドライランでコンフリクトの有無を見る
```
- **コンフリクトが無い**（"Already up to date" / クリーンにマージ）→ `git merge --abort` で戻すか、
  そのままマージを確定（ベースに追従しておくと PR がそのままマージ可能になる）。
- **コンフリクトが出た** → 解消する:
  ```bash
  git diff --check                  # コンフリクトマーカーの場所一覧
  grep -n '^<<<<<<<\|^=======\|^>>>>>>>' <file>
  ```
  - 解消は **`<<<<<<<` / `=======` / `>>>>>>>` の3マーカーを消し、正しい中身に直す**。
  - **追記ログ系（SKILL.md の「追記ログ」など、両者が末尾に行を足しただけ）は両方を残す**
    （どちらかを捨てない＝独立した知見なので）。
  - 直したら `git add <file>` → 残りを `git diff --check` で再確認（マーカーゼロ）。
  - `git commit --no-edit` でマージを確定。

## マージ後の検証（必ず）
```bash
npx vitest run        # 全テスト
npm run build         # tsc + ビルド（TS6133 など build でしか出ないエラーを拾う）
```
両方とおってから push する。落ちたらマージで持ち込んだ変更とのズレを直す。

## push と PR 作成
```bash
git push -u origin <branch>     # ネットワーク失敗時のみ 2s,4s,8s,16s で最大4回リトライ
```
- PR は GitHub MCP ツール（`mcp__github__create_pull_request`）で作る。base は `main`。
- 本文には「何を・なぜ」を簡潔に。コンフリクトを解消した場合は **その旨ひとこと**書く。
- PR を出したあと、ユーザーに「CI/レビューを watch するか」を一度たずねる
  （`subscribe_pr_activity`。CLAUDE.md の PR Activity 方針に従う）。

## 落とし穴
- **ドライランの `git merge` を放置しない**: コンフリクト解消せずに別作業を始めると、
  作業ツリーがマーカー入りのまま push される。必ず確定（commit）か `--abort` する。
- **コンフリクト解消＝勝ち負けではない**: 片方を丸ごと採用(`--theirs`/`--ours`)で済ませると
  もう片方の変更（追記ログ・別関数）が消える。中身を見て**両方の意図を残す**。
- **build を省かない**: テストが緑でも `tsc`（build）で落ちることがある（未使用 import/定数など）。
