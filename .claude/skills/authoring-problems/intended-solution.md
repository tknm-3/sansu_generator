# 意図した解き方を 強制する（プログラミング問題）

> 目的: プログラミング単元で「prompt が うたう 解き方」でしか クリアできないようにする。
> **ふつうに（まっすぐ・短く・道具を使わず）解けてしまうと 学びが うすい**。
> 設計のしかたと、それを 人手チェックに たよらず まもる **検証テスト**の早見表。

## なぜ大事か（落とし穴の実例）
- そうたい×ループ/ネストループの問題は、prompt で「コの字／かいだん／ジグザグ／四角の ふち」と
  うたっていても、盤面が `walls: []` だと **まっすぐ や L字で ショートカット**できてしまっていた
  （2026-06-03 修整）。relSolution が ゴールに着くことだけ見ても 検知できない。
- てじゅん(proc)単元は **テストが 1つも無かった**。proc_a は「てじゅんA(call) を 使わなくても
  ふつうの矢印で解ける」可能性があり、`maxSlots` でしか 暗黙に 防いでいなかった。

## 鉄則
1. **「その解き方でしか クリアできない」状態を、盤面（かべ/たから）と maxSlots で つくる。**
2. **それを まもる テストを `adventure.test.ts` に常設する**（設計ルール＝必ずテスト）。
3. prompt の ことばと 実際に 強制される 解き方を 一致させる。

## 単元別・強制のしかた と テスト

| 単元(kind) | ねらった解き方 | 強制のしかた | まもるテスト |
|---|---|---|---|
| 矢印ならべ(undefined) | 最短で ならべる | ― | `solve()===optimal`（最短一致）|
| くりかえしの たに(`loopOnly`) | ループ箱だけ | 1マス矢印を 隠す＋同じ向き2〜5連続の道 | `loopOnly` は ループ箱(2〜5)で解ける |
| もしも分岐(`branch`) | 正しい穴埋め1とおり | 4方向×穴を 総当りして 一意解になる壁 | 穴の くみあわせで 正解1とおりだけ |
| そうたい(`relative`,ループ無) | 最短の むき操作 | ― | `optimal===solveRelative()`（最短一致）|
| そうたい×ループ(`relative`+`allowLoop`) | コの字/かいだん/ジグザグ/周回 | **形の外を かべで うめて 通路にする** | **ショートカット不可**（下記）|
| てじゅん proc_a(`procMainSolution`) | てじゅんA(call)で まとめる | **矢印だけの最短 > maxSlots** にする | call なしでは maxSlots に おさまらない |
| てじゅん proc_b(`procMain`固定) | 中身(body)を 当てる | main固定で 中身しか いじれない | 中身で クリア＋optimalより短い中身は不可 |

## 「ショートカット不可」のしくみ（そうたい×ループ）
道の **まがりかど数**（すすむ向きが かわる 回数）で 判定する。
- `relSolution`（ねらった形）の まがりかど数 = `needed`。
- `solveRelative()`（最短解）の まがりかど数 = `shortest`。
- **`shortest >= needed` を 確認**。まっすぐ(0)や L字(1)で 抜ける穴があると `shortest` が
  小さくなり テストが 落ちる＝かべが たりない。
- かべの 置き方:
  - **かいだん/ジグザグ**: 形の外（off-path）のマスを **ぜんぶ かべ** にして 通路にする。
  - **コの字/Z字**: 直進する 1マスを ふさぐだけでも 直進を 封じられる（最少の壁でよい）。
  - **四角の周回(ネストループ)**: なかみ(interior)を ぜんぶ かべ にして ふちだけ 残す。
  - **gem(たから)を 形の角・経路上に置く**と、それだけで 形を 強制できることも多い
    （gem付き proc/relative は gem が 強制ずみ）。

## proc(てじゅん)の しくみ
- proc_a（中身固定・main を つくる）: パレットに **ふつうの矢印も 出る**ので、call を 使わず
  矢印だけで 解けると 学びが うすい。**矢印だけの最短手数(`solveRelative`) > maxSlots** に すれば、
  call で まとめないと スロットに 入りきらない＝てじゅん必須。
- proc_b（main固定・中身を きめる）: main が 固定で 中身しか いじれないので proc使用は 自動で強制。
  気をつけるのは **optimalより 短い中身で クリアできない**こと（短い中身で 当たると 考えずに解ける）。
  全 body（`forward/turn_right/turn_left` の 長さ1..optimal-1）を 総当りして クリア0件を 確認する。

## 新しく 形/単元を 足すとき
1. prompt の 解き方を 1つに きめる。
2. その解き方**以外で クリアできない**よう 盤面(かべ/gem)と maxSlots を 設計する。
3. **`adventure.test.ts` に「意図した解き方でしか解けない」テストを足す**（上の表のどれかに ならう）。
4. `optimal` は 手計算せず テストの `expected <実際> to be <設定値>` から 写す。
